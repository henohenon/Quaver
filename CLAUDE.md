# CLAUDE.md

このファイルは、このリポジトリで Claude Code や類似の開発エージェントが安定して作業するためのガイドです。  
判断に迷う場合は、既存コード、`package.json`、`tsconfig.json`、ユーザーの最新指示を優先してください。

## プロジェクト概要

- TypeScript プロジェクトです。
- ランタイムおよびパッケージマネージャーは Bun を優先します。
- ソースコードは `src/` 配下に置きます。
- ビルド成果物は `dist/` 配下に出力します。
- 現在のビルドコマンドは `bun run build` です。

## 基本方針

- 小さく、明確で、後から読み返しやすい変更を行ってください。
- 既存の設計、命名、ファイル配置、依存関係の使い方を尊重してください。
- ユーザーが明示していない大規模なリファクタ、フォーマット一括変更、依存関係の入れ替えは避けてください。
- 動くことだけでなく、型安全性、境界条件、失敗時の挙動、将来の保守性を確認してください。
- 不明点が実装の正しさに影響する場合は、推測で広げず、必要最小限の確認をしてください。

## 作業前の確認

作業を始める前に、必要に応じて次を確認してください。

- `package.json`: 利用可能な scripts、依存関係、モジュール方式
- `tsconfig.json`: strict 設定、出力先、module/target
- 変更対象周辺の既存コード
- 既存の README、設計メモ、テスト、設定ファイル
- `git status`: ユーザーまたは他エージェントの未コミット変更

未コミット変更がある場合、自分が触っていない変更を巻き戻したり上書きしたりしないでください。

## Bun / TypeScript のコマンド

このプロジェクトでは、Node/npm より Bun を優先してください。

- 依存関係のインストール: `bun install`
- 開発実行: `bun src/index.ts`
- ビルド: `bun run build`
- TypeScript の型チェック: `bunx tsc --noEmit`

`package.json` に `test`、`lint`、`format` などの scripts が追加されている場合は、それらを優先して使ってください。

例:

```bash
bun run test
bun run lint
bun run format
```

存在しない script を前提にしないでください。実行前に `package.json` を確認してください。

## コーディング規約

- TypeScript の `strict` を前提に、型エラーを残さないでください。
- `any` は最後の手段です。使う場合は理由が説明できる場所に限定してください。
- 公開 API、境界層、外部入力を扱う箇所では型とバリデーションを明確にしてください。
- 関数は単一の責務を持たせ、複雑な条件分岐は名前付き関数や早期 return で読みやすくしてください。
- 例外、`null`、`undefined`、空配列、空文字、失敗した I/O などの扱いを明確にしてください。
- 不要な抽象化は避け、重複が実害を持つ段階で整理してください。
- コメントは「なぜそうしているか」を補う場合に使い、「何をしているか」の説明だけのコメントは避けてください。

## ファイル構成の目安

- アプリケーションの入口: `src/index.ts`
- 主要な実装: `src/` 配下に責務ごとに分割
- テストを追加する場合は、既存の慣習がなければ対象ファイルの近く、または `test/` / `tests/` 配下に配置
- ビルド出力の `dist/` は手編集しない

新しいディレクトリ構成を導入する場合は、既存規模に対して過剰にならないようにしてください。

## 依存関係

- 標準 API と既存依存で十分な場合、新しい依存を追加しないでください。
- 依存を追加する場合は、保守状況、型定義、サイズ、代替案を考慮してください。
- Bun で動作することを確認してください。
- ロックファイルがある場合は、依存変更と一緒に更新してください。

## テストと検証

変更後は、影響範囲に応じて検証してください。

最低限:

```bash
bun run build
```

型だけを確認したい場合:

```bash
bunx tsc --noEmit
```

テスト環境が追加されている場合:

```bash
bun run test
```

検証できなかった場合は、理由と残るリスクを明確に伝えてください。

## エラー処理

- 失敗し得る処理では、呼び出し側が扱える形でエラーを返すか、適切な例外として扱ってください。
- 外部入力、ファイル、環境変数、ネットワーク、時刻に依存する処理は、失敗や欠損を想定してください。
- エラーメッセージは、原因の切り分けに役立つ具体性を持たせてください。
- ログに秘密情報、トークン、個人情報を出力しないでください。

## セキュリティ

- `.env`、秘密鍵、トークン、認証情報をコミットしないでください。
- 外部入力をそのままコマンド、SQL、パス、URL、HTML に渡さないでください。
- ファイルパスを扱う場合は、意図しないディレクトリへのアクセスやパストラバーサルを避けてください。
- 開発用の緩い設定を本番想定のコードに混ぜないでください。

## Git と変更管理

- 変更はリクエストされた範囲に限定してください。
- ユーザーの未コミット変更を巻き戻さないでください。
- 生成物、ロックファイル、設定ファイルの変更は必要性を確認してください。
- コミットを作る場合は、ユーザーから明示的に依頼されたときだけ行ってください。

コミットメッセージを書く場合は、次のような簡潔な形式を使ってください。

```text
feat: add user-facing capability
fix: handle missing configuration
chore: update project configuration
docs: update development guidance
```

## エージェントへの期待

- まず読んで理解し、次に小さく変更し、最後に検証してください。
- 変更理由、検証内容、残った懸念を簡潔に報告してください。
- 仕様が曖昧な場合は、最小の合理的な仮定を置き、その仮定を明示してください。
- 大きな変更では、実装前に短い作業計画を示してください。
- ユーザーが求めていない周辺改善を見つけても、勝手に混ぜず、必要なら別提案にしてください。
